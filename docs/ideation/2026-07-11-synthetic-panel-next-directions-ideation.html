synthetic-panel: the strongest directions after v1

v1 shipped a zero-key, MIT-licensed synthetic customer panel that ranks copy variants with a local SSR scoring instrument. This refresh looks at what to build next, grounded in the shipped code and the mid-2026 market it now sits in.

Codebase Context

The arc. synthetic-panel began inside StartupBros as an internal SSR tester (the old analytics/synthetic/ batch tooling), then was extracted and open-sourced in June 2026 as a generic, MIT, zero-API-key tool. v1 is deliberately narrow: two CLI subcommands, emit-prompts and score-reactions. The caller's own AI agent generates the persona reactions (wired through the ce-synthetic-panel-beta skill); the package owns only the deterministic prompt handoff and the local scoring. It is positioned for relative ranking ("which variant resonates more"), with absolute 1-5 scores labeled directional.

How the instrument works. Each reaction is embedded (local all-MiniLM-L6-v2 by default) and compared by cosine similarity to ordered Likert anchor statements, following Semantic Similarity Rating (arXiv:2510.08338): subtract the minimum similarity, normalize to a distribution, apply a power-law temperature, average across anchor sets, then take a weighted 1-5 score. A comparability guard flags near-duplicate samples so a collapsed run is not trusted.

Notable state and leverage points (from the shipped code).

  • The ranking sorts on point scores alone; the full per-sample distributions are computed and then discarded for ranking, and comparability is an all-or-nothing verdict driven by any single issue.
  • Per-variant scores are averaged across personas into one number, so segment disagreement (a founder loves it, a learner shrugs) is invisible in the headline.
  • The comparability guard only catches near-duplicate samples; it does not flag sycophantic, generic, or caricatured reactions.
  • The default anchor file ships three sets while the scoring docstring describes six, and one intent-to-act ladder is reused for all six stimulus types (headline, email subject, offer, value prop, CTA, price).
  • A large body of API-only machinery (synchronous, async, and Anthropic batch test runners, plus a CalibrationStore) is unreachable from the two-subcommand CLI and depends on a calibration module that does not ship, so it is inert in v1.

External context (mid-2026 landscape). The synthetic-research category is crowded and richly funded (one competitor-authored 2026 market map tallies over $1.5B in venture capital), but every notable player (Synthetic Users, Aaru, Simile, Yabble/YouGov, Qualtrics Edge, Artificial Societies) is closed-source SaaS priced per seat or per interview, and most now advertise absolute calibration accuracy of 85 to 95 percent. That is precisely the claim the research literature disputes: Bisbee et al. (2024) show synthetic panels narrow variance while inflating point estimates, Sarstedt et al. (2024) find only about a quarter of 285 silicon-versus-human comparisons "similar," and Samoylov (2024) showed reworded prompts swinging a persona's mean income from $111k to $272k. The SSR paper itself validates that the method recovers the relative ranking of concepts, which is exactly the narrow claim v1 makes. No open, vendor-neutral tool or leaderboard turned up in this refresh's search; open benchmarks that could anchor validation now exist (Twin-2K-500, the A/B-holdout SimGym, ANES/WVS backtesting). On the instrument itself, all-MiniLM-L6-v2 carries a known training-data commercial gray area, and clean MIT/Apache successors (BGE-M3, Qwen3-Embedding-0.6B) are now laptop-runnable. Demand from indie builders and CRO practitioners is real but bounded: they want a cheap, honest pre-filter, and their loudest complaint is exactly the overconfidence a relative-ranking tool is built to avoid.

Several landscape sources were reached through search snippets and secondary citations rather than direct fetches, and vendor accuracy figures are self-reported. External bases below name their source so each can be spot-checked before it is leaned on.

Topic Axes

  • Calibration and trust: turning directional scores into a signal a user can act on: uncertainty, significance, and validation against real outcomes.
  • Scoring instrument: the SSR anchors, the embedding model, the distribution math, the comparability logic.
  • Generation seam: how reactions are produced and how independent and realistic they are, plus the zero-key handoff and the dead API paths behind it.
  • Persona realism: the persona schema, its depth and provenance, and the segment structure the panel represents.
  • Adoption and distribution: install friction, who can run it without an agent, and where it fits into a recurring workflow.

I-1 Uncertainty-aware ranking: report the tie, not just the winner

Calibration and trust Confidence 85% Complexity Medium

The tool already computes a full 1-5 probability distribution for every sample and every variant, then throws that richness away: the ranking sorts purely on the point score, so two variants at 3.81 and 3.79 are presented as first and second place when the gap is noise. Add an uncertainty layer on top of the numbers the instrument already produces. Use the per-sample spread and distribution overlap to label each pairwise comparison as a clear difference, a lean, or a statistical tie, and give the user a "you would need N more samples to separate these" hint. In the same pass, replace the all-or-nothing comparability verdict with a graded one that scopes trust to the clean persona-variant cells instead of failing the whole run on one issue.

Basis
direct The ranking is built by sorting per-variant point scores only (testing.py, the ranking comprehension), while the averaged distribution is computed alongside and left unused for the verdict; comparability is "not_comparable" if issues else "comparable", so one missing sample condemns the whole run.
external The category's central credibility problem is overstated confidence: Bisbee et al. (Political Analysis, 2024) show synthetic panels narrow variance, and Samoylov (Conjointly, 2024) shows outputs swinging wildly with prompt wording. A tool that openly reports "this is a tie" is the direct antidote and matches its own relative-ranking promise.
Rationale

The product's one-line promise is "which variant resonates more." Right now it answers that question even when it should not. Distinguishing a real difference from sampling noise is the single change that most raises trust, and it is the feature every closed competitor avoids because it undercuts their absolute-accuracy marketing. It costs almost nothing in new machinery because the distributions already exist.

Downsides

Bootstrap or overlap statistics on n=2 samples are themselves shaky; the honest version nudges users toward more samples, which costs them generation time. Presenting ties can feel less satisfying than a clean leaderboard.

I-2 Segment-disagreement view: where the personas split

Persona realism Confidence 88% Complexity Low

Scores are computed per persona-variant, then averaged into one number per variant before ranking. That average is where the most actionable signal disappears: the interesting result is rarely "variant C wins overall," it is "variant C wins decisively with founders and actively loses with first-time learners." Surface a compact persona-by-variant matrix and call out the disagreements explicitly: the variant with the highest floor, the one with the widest spread, and any variant that flips rank between segments. The per-persona data is already retained; this is a presentation and small-analysis change, not new scoring.

Basis
direct Per-variant scoring averages the persona scores (testing.py, the per-variant aggregation divides the summed group scores by the group count) even though per_persona_variant retains every cell; the headline ranking never exposes it.
Rationale

Marketers do not ship copy to an "average" persona; they choose who to win and who to sacrifice. Disagreement is the decision-grade output, and it turns a one-number answer into a segmentation insight that no averaged leaderboard can give. It is close to free given the retained data.

Downsides

With few personas the matrix is thin and can imply more precision than n=2 samples support; it pairs best with I-1 so cells are labeled by confidence. More surface area to render in a CLI-first tool.

I-3 Stimulus-aware anchor sets (and finish the six)

Scoring instrument Confidence 80% Complexity Low

The instrument measures one latent dimension, intent to act ("I would take the next step"), and applies that single ladder to all six stimulus types. But a price test is not an intent test: the useful axis for pricing is "feels fair versus feels expensive," for a headline it is "clear versus confusing," for a value prop it is "relevant versus generic." Ship stimulus-appropriate anchor ladders so the score measures the thing that actually matters for each format. In the same pass, reconcile the default anchor data with the method: the file ships three sets while the SSR docstring calls for six, and the paper's variance reduction depends on that averaging.

Basis
direct anchors.default.yaml contains set_1 through set_3, all phrased as intent to act, while scoring.py documents "averaging across 6 distinct reference statement sets"; get_stimulus_prompt in config.py defines six distinct stimulus types that all score against the one ladder.
external The SSR paper (arXiv:2510.08338) frames the anchors as the elicited construct, and its validity is conditional on measuring the right latent scale; using an intent ladder to score a fairness question is a construct mismatch the method itself warns against.
Rationale

This is the cheapest way to make the scores more valid rather than just more numerous. Anchor sets are data, not code, so it invites contribution and keeps the instrument inspectable, which is the open project's core differentiator against closed panels.

Downsides

Divergent per-type ladders make cross-format comparisons less meaningful (arguably correct, but a behavior change). More anchor sets to author and validate against the "negative below neutral below positive" ordering test the repo already enforces.

I-4 A validation harness that measures the panel against real outcomes

Calibration and trust Confidence 70% Complexity High

The README calls calibration "planned future work," and the codebase already gestures at it: prediction tracking and a CalibrationStore are scaffolded but inert because the module never ships. Make the loop real and honest. Ship a small labeled benchmark (a set of copy variants with a known preference order from real A/B results or human ratings) and a validate command that reports how well the panel's ranking reproduces the known order (rank correlation), not an absolute accuracy claim. Then make it a flywheel: let users opt in to attach their own real outcomes to stored predictions, so the tool accumulates evidence about when its rankings hold and when they do not.

Basis
direct track_predictions and get_cached_result both do from calibration import CalibrationStore inside try/except ImportError (testing.py, config.py), and no calibration module ships, so the calibration path is dead. The README lists benchmarking as future work.
external Vendor-neutral public harnesses now exist to model this on: Twin-2K-500 (arXiv:2505.17479) and the A/B-holdout SimGym (2026); ANES/WVS backtesting is the standard academic method. No open leaderboard for "does this panel predict real preferences" surfaced in this search, which points to open whitespace an OSS tool is well placed to claim.
Rationale

This is the idea that converts the project's honesty from a disclaimer into a measurement. "We only claim relative ranking, and here is the harness that shows how well our rankings hold" is a stronger and more defensible position than the 85 to 95 percent absolute-accuracy claims competitors make without publishing their math. It also gives the inert scaffolding a real purpose or a clean removal.

Downsides

The hard part is data: a credible benchmark needs real labeled outcomes, which are scarce and often proprietary; a weak benchmark is worse than none. Opt-in outcome collection raises privacy and incentive questions and must never become silent telemetry.

I-5 Persona provenance: ground the panel in real evidence

Persona realism Confidence 72% Complexity Medium

Every score the panel produces inherits the realism of the personas behind it, and today those personas are invented. The shipped examples are demographics only, and even the richer schema is a fixed list of asserted traits (motivations, fears, messaging triggers) with no record of where any of them came from. That is exactly the input the silicon-sampling critiques blame for caricature. Let personas carry provenance instead: seed traits from real evidence the user already has (a handful of customer reviews, support tickets, sales-call notes, survey verbatims), and record, per trait, which snippet it came from. The panel then reasons from grounded, auditable personas, and a reviewer can see why a persona holds a belief rather than taking it on faith.

Basis
direct The shipped personas are demographics only (personas.example.yaml: name, age, location, family, occupation, income), and the rich schema (config.py required fields for motivations, psychology, fears, messaging triggers) is a fixed set of asserted fields with no provenance; build_persona_prompt injects them into the prompt verbatim as fact.
external Interview- and data-grounded personas outperform demographic-only prompting on fidelity (Twin-2K-500, arXiv:2505.17479, reaches roughly 72 to 88 percent of the human test-retest ceiling), while the central critique of synthetic panels (Bisbee et al. 2024, plus the mode-collapse literature) is that ungrounded persona conditioning yields caricature and narrowed variance. Grounding is the lever that most moves realism.
Rationale

Grounding personas in the user's own evidence is the highest-leverage way to make reactions less generic and less sycophantic at the source, upstream of I-6's post-hoc quality gate. It also plays to the open tool's strength: the evidence stays local and inspectable, so realism becomes something a user can audit rather than a claim they have to trust.

Downsides

Users may not have clean evidence to hand, and garbage in still applies; grounding raises the setup burden the zero-friction path tries to minimize, so it pairs with I-8's easier run. Provenance tracking adds schema and interface surface, and pasted evidence can carry personal data that must never leave the user's machine.

I-6 A reaction-quality gate for sycophancy and mode collapse

Generation seam Confidence 72% Complexity Medium

The panel embeds whatever reaction text the agent produced and treats it as ground truth. The one guard that exists only catches near-duplicate samples after the fact; it says nothing about the failure mode the whole field complains about, which is sycophantic, generic, or caricatured reactions that inflate every score toward the top of the scale. Add a pre-scoring quality gate: flag reactions that are too short on specifics, that echo the stimulus wording, that cluster suspiciously close to a "generic positive" reference, or that are near-identical across genuinely different variants. Surface these as warnings that scope trust, in the same spirit as the existing comparability verdict.

Basis
direct The collapse detector compares samples on score_spread and embedding_distance floors (testing.py) to catch duplicates only; there is no check that a reaction is specific, non-sycophantic, or discriminating between variants before it is scored.
external This is the most consistent complaint in the field: NN/g finds synthetic users skew "sycophantic," indie builders report pure-LLM feedback "tends to be a bit overoptimistic," and the mode-collapse literature (Paglieri et al.) shows even explicit diversity prompts collapse to a stereotype. Practitioners already hand-build "skeptical researcher" personas to fight this.
Rationale

Cheap positivity is the exact way a synthetic panel silently lies. A gate that flags it, before the numbers are computed, protects the credibility of every downstream score and directly targets the objection reviewers and skeptics raise first.

Downsides

Heuristics for "sycophantic" are fuzzy and risk false positives that annoy users; a too-aggressive gate could reject legitimately enthusiastic reactions. Some of this belongs upstream in the generation prompt, not only as a post-hoc filter.

I-7 "Why it ranked": copy attribution by ablation

Scoring instrument Confidence 60% Complexity Medium-High

A ranking tells a user which variant won; it does not tell them what to change. Attribution closes that gap: take a variant, make targeted edits (drop the proof clause, swap the urgency phrase), run each edited version back through the panel, and read off which change moved the resonance score. The output becomes a copy edit, not just a number: "the proof clause is carrying this headline; the urgency phrase is doing nothing." One honest caveat drives the whole design: the panel scores the persona's reaction, not the copy directly, so an edited variant needs a freshly generated reaction, which is an agent round-trip, not a free local operation. That means attribution is a targeted sweep of a few candidate edits, run through the pipeline the tool already has, not an exhaustive word-by-word ablation.

Basis
direct The seam already supports this shape: emit-prompts takes a variants file and score-reactions is a pure function of reaction text with cached local embeddings, so ablated copy is just additional variants through the existing emit, generate, score loop. The local scoring half is effectively free; only the generate step per edit has real cost.
reasoned Attribution by ablation is a standard, well-understood technique; the novelty is applying it to turn a resonance score into an editable copy diff, which is the actionable artifact marketers actually want. Because each edit costs one generation round-trip, a handful of hypothesis-driven edits (not a full ablation grid) is the affordable and useful version.
Rationale

It moves the product up the value chain from "score my copy" to "improve my copy," a far stickier promise and a natural differentiator no closed leaderboard offers. It compounds with I-1 and I-3 because an attribution is only trustworthy once ties are detected and the anchors measure the right construct.

Downsides

Each ablation needs a regenerated reaction, so cost scales with the number of edits and this is agent work, not just local scoring; an exhaustive ablation grid is out of reach. Attribution to a specific clause can imply more precision than a noisy, small-sample instrument supports, so it must be framed as directional, and it is only as good as the generation faithfully reflecting the edit.

I-8 One-command run: an optional local generator, and honest cleanup

Adoption and distribution Confidence 75% Complexity Medium

v1 deliberately delegated generation to the caller's agent, which is right for the skill workflow but leaves two problems. First, a user without an agent harness cannot run the loop end to end: they must hand-shuttle a reactions.json between emit-prompts and score-reactions. Second, a large body of API-only run machinery (sync, async, and Anthropic batch runners, plus the calibration store) sits in the tree but is unreachable and non-functional. Resolve both together: add an optional local generation provider (Ollama or llama.cpp) so synthetic-panel run can do emit, generate, and score in one call for the zero-agent user, and in doing so either wire or delete the dead API paths so the codebase honestly reflects what it does.

Basis
direct The CLI wires only emit-prompts and score-reactions (cli.py subparsers); run_test, run_test_async, and run_test_batch call get_llm_client, which raises NotImplementedError("generation is provided by the skill in v1") (testing.py), and the batch path plus the calibration store are unreachable and non-shipping.
external No comparable zero-API-key, local-scoring OSS tool surfaced on GitHub in this search (the adjacent projects are persona datasets, not scorers), and indie builders repeatedly ask for a cheap, self-contained pre-filter. A one-command local run removes the last dependency between the tool and a non-agent user.
Rationale

Zero-key is the headline promise; zero-agent would make it true for everyone, not just Claude Code users, which widens adoption sharply. Whichever way the generation question is answered, the dead code has to be resolved: shipping runners that cannot run erodes trust in an open codebase whose whole pitch is inspectability.

Downsides

Bundling or depending on a local generator reintroduces a heavy model download and quality variance the zero-key scoring path deliberately avoided; small local models may generate weak reactions that make the panel look bad. Pruning has to be careful not to remove a path someone downstream relies on. This is the largest and most opinionated bet in the set.
